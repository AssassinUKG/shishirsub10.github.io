![adminer](/assets/images/htb-boxes/adminer.png)

Adminer is an easy linux box by [polarbearer](https://www.hackthebox.eu/home/users/profile/159204) & [GibParadox](https://www.hackthebox.eu/home/users/profile/125033). First a hidden directory was found on robots.txt entry which contained files with username and passwords, which I used to login on port 21 and a backup file was found on the ftp port containing more sensitive information. Then adminer.php was found inside /utility-scripts after using gobuster, which was found vulnerable and exploited to get latest credential for waldo which I used to login using ssh. In the box there was a sudoers entry which is exploited with python library hijacking to get a root shell on the box.

### Flags used
*   dir for specifying in directory bruteforcing mode as gobuster also supports dns and vhost bruteforcing
*   -w to specify the wordlist to use for bruteforcing
*   -u to specify the url
*   -x to specify the extensions for filename. I have used txt as the credentials might be stored on .txt file and php just to show that you could bruteforce for files with  multiple extension
*   -t for threads to speed up the process. ie default is 10

_directory-list-2.3-medium.txt_ wordlist comes by default in most pentetration testing distros. If you don't have any wordlists on your box, you can clone the [SecLists](https://github.com/danielmiessler/SecLists) repo from github using

### Adminer.php

![2](/assets/images/adminer/2.png)

Adminer (formerly phpMinAdmin) is a full-featured database management tool written in PHP. And looks like the currently running version of our Adminer is 4.6.2.


I looked around and found a [post](https://www.foregenix.com/blog/serious-vulnerability-discovered-in-adminer-tool) explaining that this version of Adminer is vulnerable and we could pull the local files from the remote server.
To exploit this first we have to connect the adminer to our local mysql server and then we can pull the files from the remote server into our local database.

## Exploiting the vulnerability

Now that we have setup a database and a user, we have to change the mysql configuration as mysql only listens on local interface ie 127.0.0.1. So we have to tell mysql server to listen on all interfaces.


We have to change this bind-address value to be 0.0.0.0

```console
local@local:~/Desktop/htb/adminer# cd /etc/mysql/
local@local:/etc/mysql# grep -R 0.0.0.0

```

# Privilege Escalation
Before running any scripts like [linpeas](https://github.com/carlospolop/privilege-escalation-awesome-scripts-suite) or [LinEnum](https://github.com/rebootuser/LinEnum) for checking the potential privilage escalation vector, I like to do a manual enumeration first.

### Sudo -l
```console
waldo@admirer:~$ sudo -l
[sudo] password for waldo: 
Matching Defaults entries for waldo on admirer:
    env_reset, env_file=/etc/sudoenv, mail_badpass, secure_path=/usr/local/sbin\:/usr/local/bin\:/usr/sbin\:/usr/bin\:/sbin\:/bin, listpw=always

User waldo may run the following commands on admirer:
(ALL) SETENV: /opt/scripts/admin_tasks.sh
```

Looking at the entry, any user on the box can run  `/opt/scripts/admin_tasks.sh` with sudo privilages.

### Content of backup.py
```console
waldo@admirer:/opt/scripts$ cat backup.py 
#!/usr/bin/python3

from shutil import make_archive

src = '/var/www/html/'

# old ftp directory, not used anymore
#dst = '/srv/ftp/html'

dst = '/var/backups/html'

make_archive(dst, 'gztar', src)
```
Looking at the content of _backup.py_, `make_archive` is imported from file _shutil.py_. If we can create a file called **shutil.py** with function **make_archive** on the same directory, then it will go and import the function from the file on the same directory as **backup.py**. This concept is called library hijacking in python.
